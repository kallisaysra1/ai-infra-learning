# Exercise 03: Build a Reusable Terraform Module Library

**Duration:** 3 hours
**Difficulty:** Intermediate
**Prerequisites:** Exercise 01; lab 01-02

## Objective

Build an internal Terraform module library: 3 well-designed modules (VPC, EKS, RDS) with versioning, documentation, examples, and tests. Publish to a private registry so other teams consume them via `source = "..."` rather than copy-paste.

## Why this matters

Without a module library, every team writes their own VPC. Drift accumulates, security gaps proliferate, and onboarding takes weeks. With one, every new project is `module "vpc" { source = "company/vpc"; version = "~> 2" }`.

## Requirements

1. Three modules in a single repo with consistent layout.
2. Semantic versioning, tagged releases.
3. `terraform-docs` auto-generated documentation per module.
4. At least one **example** per module that's runnable end-to-end.
5. **Terratest** (Go) or **Kitchen-Terraform** for integration tests.
6. CI workflow: lint + format + docs + tests on every PR.

## Step-by-step

### Step 1 — Repo layout (15 min)
```
terraform-modules/
├── vpc/
│   ├── main.tf, variables.tf, outputs.tf, versions.tf
│   ├── README.md (auto-generated)
│   ├── examples/
│   │   └── basic/
│   └── test/
│       └── vpc_test.go
├── eks/
├── rds/
└── .github/workflows/test.yml
```

### Step 2 — Module conventions (30 min)

Each module has:
- `versions.tf`: required Terraform + provider versions.
- `variables.tf`: every input with description, type, optional default + validation.
- `outputs.tf`: every output with description.
- `main.tf`: resources + locals.
- `README.md`: auto-generated by `terraform-docs`.

Validation example:
```hcl
variable "cidr_block" {
  type        = string
  description = "VPC CIDR block"
  validation {
    condition     = can(cidrnetmask(var.cidr_block))
    error_message = "Must be a valid CIDR."
  }
  validation {
    condition     = tonumber(split("/", var.cidr_block)[1]) <= 22
    error_message = "VPC CIDR must be /22 or larger to fit subnet plan."
  }
}
```

### Step 3 — VPC module (45 min)
Per lab 01. Add module-quality polish: tagging, optional NAT count, multi-AZ.

### Step 4 — EKS module (45 min)
Per lab 02. Add: managed node groups via variables, configurable IRSA support, outputs for OIDC provider URL.

### Step 5 — RDS module (30 min)
Postgres with: multi-AZ option, automated backups, parameter group, encryption-at-rest, output for the credentials secret ARN.

### Step 6 — terraform-docs (15 min)
```bash
brew install terraform-docs
echo 'formatter: "markdown table"' > .terraform-docs.yml
terraform-docs markdown table --output-file README.md --output-mode inject vpc/
```
README auto-includes inputs/outputs tables. Re-run in CI; fail PR if README stale.

### Step 7 — Examples (15 min)
```
vpc/examples/basic/main.tf:
  module "vpc" {
    source = "../../"
    name   = "example"
    cidr_block = "10.0.0.0/16"
    azs        = ["us-west-2a", "us-west-2b"]
  }
```
This is the "how to use it" reference + can be terraform-applied as a smoke test.

### Step 8 — Terratest (30 min)
```go
// vpc/test/vpc_test.go
package test

import (
    "testing"
    "github.com/gruntwork-io/terratest/modules/terraform"
    "github.com/stretchr/testify/assert"
)

func TestVPC(t *testing.T) {
    opts := &terraform.Options{
        TerraformDir: "../examples/basic",
        NoColor: true,
    }
    defer terraform.Destroy(t, opts)
    terraform.InitAndApply(t, opts)
    
    vpcId := terraform.Output(t, opts, "vpc_id")
    assert.Regexp(t, "^vpc-", vpcId)
}
```

### Step 9 — Versioning + tagging (15 min)
SemVer per module. Tag the repo: `vpc/v1.0.0`, `eks/v1.0.0`, etc. Or one global tag per release.

Consumers pin:
```hcl
module "vpc" {
  source  = "git::ssh://git@github.com/company/terraform-modules.git//vpc?ref=vpc/v1.0.0"
}
```

### Step 10 — CI workflow (15 min)
```yaml
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3
      - run: terraform fmt -check -recursive
      - run: for d in vpc eks rds; do terraform -chdir=$d init -backend=false; terraform -chdir=$d validate; done
      - run: terraform-docs markdown table vpc/ | diff - vpc/README.md
  test:
    needs: validate
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
      - run: cd vpc/test && go test -timeout 30m
```

## Deliverables

1. Three modules in the agreed layout.
2. Auto-generated READMEs.
3. Runnable examples.
4. Terratest passing for at least the VPC module.
5. CI workflow validating every PR.

## Validation

- [ ] `terraform validate` succeeds in each module.
- [ ] `terraform-docs` output matches README.
- [ ] Example applies + destroys cleanly.
- [ ] Terratest passes.

## Stretch goals

- Publish to a **private Terraform registry** (Terraform Cloud, Spacelift, or self-hosted).
- Add **module composition**: a higher-level `ml-platform` module composing VPC + EKS + RDS.
- Add **policy gates** (Sentinel / OPA) to enforce that no team can deploy without these modules.

## Common pitfalls

- **Modules tightly coupled to one account** — Hardcoded ARNs / account IDs. Always parameterize.
- **No versioning** — Consumers track main; breakage on every commit. Always tag.
- **Documentation drift** — Auto-generate; otherwise nobody updates it.
